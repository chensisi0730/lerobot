# 异步推理

通过我们的 [SmolVLA](https://huggingface.co/papers/2506.01844)，我们引入了一种在真实世界机器人上运行推理的新方法，**将动作预测与动作执行解耦**。
在本教程中，我们将展示如何使用异步推理（_async inference_）来运行微调版本的 SmolVLA，以及 LeRobot 支持的所有策略。
**尝试使用 LeRobot 支持的所有策略进行异步推理！**

**你将学到：**

1. 异步推理为何重要，以及它与更传统的顺序推理的比较方式。
2. 如何在同一台机器上启动 `PolicyServer` 并连接 `RobotClient`，甚至跨网络连接。
3. 如何为你的机器人和策略调整关键参数（`actions_per_chunk`、`chunk_size_threshold`）。

如果遇到困难，可以加入我们的 [Discord 社区](https://discord.gg/s3KuuzsPFb)！

简而言之：使用 _异步推理_，当策略服务器已经在计算下一块动作时，你的机器人持续动作——消除"等待推理"的延迟，解锁更流畅、反应更灵敏的行为。
这与同步推理（sync）根本不同，在同步推理中，机器人在策略计算下一块动作时保持空闲。

## 开始使用异步推理

你可以在我们的[博客文章](https://huggingface.co/blog/async-robot-inference)中阅读更多关于异步推理的信息。本指南旨在帮助你快速设置并在你的环境中运行异步推理。

首先，使用 `async` 标签安装 `lerobot`，以安装运行异步推理所需的额外依赖。

```shell
pip install -e ".[async]"
```

然后，启动一个策略服务器（在一个终端中，或在另一台机器上），指定客户端连接的主机地址和端口。
你可以运行以下命令启动策略服务器：

```shell
python -m lerobot.async_inference.policy_server \
     --host=127.0.0.1 \
     --port=8080
```

这将启动一个监听 `127.0.0.1:8080`（localhost，端口 8080）的策略服务器。此时，策略服务器是空的，因为与运行什么策略以及使用什么参数相关的所有信息都是在与客户端的第一次握手期间指定的。使用以下命令启动客户端：

```shell
python -m lerobot.async_inference.robot_client \
    --server_address=127.0.0.1:8080 \ # SERVER：策略服务器的主机地址和端口
    --robot.type=so100_follower \ # ROBOT：你的机器人类型
    --robot.port=/dev/tty.usbmodem585A0076841 \ # ROBOT：你的机器人端口
    --robot.id=follower_so100 \ # ROBOT：你的机器人 id，用于加载校准文件
    --robot.cameras="{ laptop: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}, phone: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}}" \ # POLICY：用于获取帧的相机，键与策略期望的键匹配
    --task="dummy" \ # POLICY：运行策略的任务（`Fold my t-shirt`）。不一定为所有策略定义，如 `act`
    --policy_type=your_policy_type \ # POLICY：运行的政策类型（smolvla、act 等）
    --pretrained_name_or_path=user/model \ # POLICY：服务器上模型名称/路径到要运行的检查点（例如 lerobot/smolvla_base）
    --policy_device=mps \ # POLICY：服务器上运行策略的设备（cuda、mps、xpu、cpu）
    --actions_per_chunk=50 \ # POLICY：一次输出的动作数量
    --chunk_size_threshold=0.5 \ # CLIENT：发送新观测到服务器之前的块大小阈值
    --aggregate_fn_name=weighted_average \ # CLIENT：聚合重叠部分动作的函数
    --debug_visualize_queue_size=True # CLIENT：是否在运行时可视化队列大小
```

总之，你需要指定以下说明：

- `SERVER`：策略服务器的地址和端口
- `ROBOT`：连接的机器人类型、连接的端口以及机器人的本地 `id`
- `POLICY`：运行的策略类型，以及服务器上模型名称/路径到要运行的检查点。你还需要指定服务器应该使用什么设备，以及一次输出多少动作（限制为策略最大动作值）。
- `CLIENT`：发送新观测到服务器之前的块大小阈值，以及聚合重叠部分动作的函数。你还可以选择在运行时可视化队列大小，以帮助你调整 `CLIENT` 参数。

重要的是：

- `actions_per_chunk` 和 `chunk_size_threshold` 是为你的设置调整的关键参数。
- `aggregate_fn_name` 是聚合重叠部分动作的函数。你可以向函数注册表添加新函数，或者在 `robot_client.py` 中添加你自己的函数。
- `debug_visualize_queue_size` 是调整 `CLIENT` 参数的有用工具。

## 完成了！你现在应该能看到你的机器人移动了 😉

## 异步与同步推理

同步推理依赖于交错动作块预测和动作执行。这固有地导致 _空闲帧_，即机器人等待策略输出新动作块时的空闲帧。
反过来，推理受到明显实时延迟的困扰，由于缺乏可用的动作，机器人干脆停止行动。
随着机器人模型越来越大，这个问题只会变得更加严重。

<p align="center">
  <img
    src="https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/async-inference/sync.png"
    width="80%"
  ></img>
</p>
<p align="center">
  <i>同步推理</i> 在策略计算下一块动作时让机器人空闲。
</p>

为了克服这个问题，我们设计了异步推理，这是一种将动作规划和执行解耦的范式，导致（1）更高的适应性，最重要的是（2）没有空闲帧。
关键的是，使用异步推理，下一块动作在当前块耗尽 _之前_ 就被计算出来，从而没有空闲。
更高的适应性通过在重叠部分聚合不同的动作块来确保，获得最新计划和更紧密的控制循环。

## 启动策略服务器

策略服务器是围绕 `PreTrainedPolicy` 的包装器，使其与来自机器人客户端的观测进行交互。
策略服务器初始化为空容器，在机器人客户端和策略服务器之间的初始握手中填充所请求的策略。
因此，启动策略服务器就像指定主机地址和端口一样简单。如果你在与机器人客户端相同的机器上运行策略服务器，可以使用 `localhost` 作为主机地址。

```bash
python -m lerobot.async_inference.policy_server \
     --host=127.0.0.1 \
     --port=8080
```

```python
from lerobot.async_inference.configs import PolicyServerConfig
from lerobot.async_inference.policy_server import serve

config = PolicyServerConfig(
    host="localhost",
    port=8080,
)
serve(config)
```

这将监听 `localhost:8080` 以等待来自关联的 `RobotClient` 的传入连接，该客户端将在第一次客户端-服务器握手期间传达要运行的策略。

## 启动机器人客户端

`RobotClient` 是 `Robot` 实例的包装器，`RobotClient` 连接到（可能是远程的）`PolicyServer`。
`RobotClient` 将观测流式传输到 `PolicyServer`，并接收在服务器上运行推理获得的动作块（我们假设服务器比机器人控制器具有更好的计算资源）。

```bash
python -m lerobot.async_inference.robot_client \
    --server_address=127.0.0.1:8080 \ # SERVER：策略服务器的主机地址和端口
    --robot.type=so100_follower \ # ROBOT：你的机器人类型
    --robot.port=/dev/tty.usbmodem585A0076841 \ # ROBOT：你的机器人端口
    --robot.id=follower_so100 \ # ROBOT：你的机器人 id，用于加载校准文件
    --robot.cameras="{ laptop: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}, phone: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}}" \ # POLICY：用于获取帧的相机，键与策略期望的键匹配
    --task="dummy" \ # POLICY：运行策略的任务
    --policy_type=your_policy_type \ # POLICY：运行的策略类型
    --pretrained_name_or_path=user/model \ # POLICY：模型名称/路径
    --policy_device=mps \ # POLICY：服务器上的设备
    --actions_per_chunk=50 \ # POLICY：一次输出的动作数量
    --chunk_size_threshold=0.5 \ # CLIENT：块大小阈值
    --aggregate_fn_name=weighted_average \ # CLIENT：聚合函数
    --debug_visualize_queue_size=True # CLIENT：可视化队列大小
```

```python
import threading
from lerobot.robots.so_follower import SO100FollowerConfig
from lerobot.cameras.opencv import OpenCVCameraConfig
from lerobot.async_inference.configs import RobotClientConfig
from lerobot.async_inference.robot_client import RobotClient
from lerobot.async_inference.helpers import visualize_action_queue_size

# 1. 创建机器人实例
"""通过运行 `python lerobot/find_cameras.py` 检查你设置中可用的相机"""
# 这些相机必须与策略期望的相机匹配
# 查看你使用的策略在 Hub 上的 config.json
camera_cfg = {
    "top": OpenCVCameraConfig(index_or_path=0, width=640, height=480, fps=30),
    "side": OpenCVCameraConfig(index_or_path=1, width=640, height=480, fps=30)
}

robot_cfg = SO100FollowerConfig(
  port="/dev/tty.usbmodem585A0076841",
  id="follower_so100",
  cameras=camera_cfg
)

# 3. 创建客户端配置
client_cfg = RobotClientConfig(
    robot=robot_cfg,
    server_address="localhost:8080",
    policy_device="mps",
    client_device="cpu",
    policy_type="smolvla",
    pretrained_name_or_path="<user>/smolvla_async",
    chunk_size_threshold=0.5,
    actions_per_chunk=50,  # 确保这小于策略的最大动作数
)

# 4. 创建并启动客户端
client = RobotClient(client_cfg)

# 5. 指定任务
task = "Don't do anything, stay still"

if client.start():
    # 启动动作接收器线程
    action_receiver_thread = threading.Thread(target=client.receive_actions, daemon=True)
    action_receiver_thread.start()

    try:
        # 运行控制循环
        client.control_loop(task)
    except KeyboardInterrupt:
        client.stop()
        action_receiver_thread.join()
        #（可选）绘制动作队列大小
        visualize_action_queue_size(client.action_queue_size)
```

以下两个参数在每个设置中都是关键的：

| 超参数 | 默认值 | 作用 |
| ------ | ------ | ---- |
| `actions_per_chunk` | 50 | 策略一次输出的动作数量。典型值：10-50。|
| `chunk_size_threshold` | 0.7 | 当队列 ≤ 50% 满时，客户端发送新观测。值在 [0, 1] 范围内。|

<Tip>
  `actions_per_chunk` 和 `chunk_size_threshold` 的不同值确实会导致不同的行为。
</Tip>

一方面，增加 `actions_per_chunk` 的值将导致耗尽可执行动作的可能性降低，因为计算新块时将有更多可用动作。
然而，较大的 `actions_per_chunk` 值也可能导致动作不太精确，因为预测更长时间跨度的动作会导致复合误差。

另一方面，增加 `chunk_size_threshold` 的值将导致更频繁地向 `PolicyServer` 发送观测进行推理，从而导致更多更新动作块，在重要部分重叠。这导致高适应性，限制为每个观测预测一个动作块，而新动作块产生时被消耗的只是边际。
这个选项也会给推理管道带来更多压力，因为请求很多。相反，接近 0.0 的 `chunk_size_threshold` 值会折叠到同步边缘情况，只有在当前块耗尽时才发送新观测。

我们发现 `actions_per_chunk` 和 `chunk_size_threshold` 的默认值在我们为 [SmolVLA 论文](https://huggingface.co/papers/2506.01844) 所做的实验中效果很好，但建议尝试不同的值以找到最适合你设置的值。

### 为你的设置调整异步推理

1. **仔细选择你的计算资源。** [PI0](https://huggingface.co/lerobot/pi0) 在推理时占用 14GB 内存，而 [SmolVLA](https://huggingface.co/lerobot/smolvla_base) 只需要约 2GB。你应该识别最适合你使用情况的计算资源，记住较小的策略需要更少的计算资源。策略和所使用设备的组合（CPU 密集型，使用 MPS，或给定 NVIDIA GPU 上的 CUDA 核心数量）直接影响你应该期望的平均推理延迟。
2. **根据推理延迟调整你的 `fps`。** 当服务器生成新动作块时，客户端并不空闲，而是正在执行其当前动作队列。如果两个进程以根本不同的速度发生，客户端可能会耗尽队列。因此，如果你的队列中持续没有动作，你应该减少 fps。
3. **调整 `chunk_size_threshold`**。
   - 接近 `0.0` 的值导致几乎顺序的行为。接近 `1.0` 的值 → 每步都发送观测（更多带宽，依赖于好的世界模型）。
   - 我们发现 0.5-0.6 左右的值效果很好。如果你想调整这个，启动一个 `RobotClient` 并将 `--debug_visualize_queue_size` 设置为 `True`。这将在运行时绘制动作队列大小演变，你可以用它来找到最适合你设置的 `chunk_size_threshold` 值。

## 结论

异步推理代表了实时机器人控制的重要进展，解决了长期以来困扰机器人应用的推理延迟这一根本挑战。通过本教程，你已经学会了如何实现一个完整的异步推理管道，消除空闲帧并实现更流畅、反应更灵敏的机器人行为。

**要点：**

- **范式转变**：异步推理将动作预测与执行解耦，允许机器人在并行计算新动作块时持续动作
- **性能优势**：消除了同步方法固有的"等待推理"延迟，随着策略模型越来越大变得越来越重要
- **灵活架构**：服务器-客户端设计支持分布式计算，推理可以在强大的远程硬件上运行，同时保持实时机器人控制
- **可调参数**：成功取决于为你的特定硬件、策略和任务需求正确配置 `actions_per_chunk` 和 `chunk_size_threshold`
- **通用兼容性**：适用于 LeRobot 支持的所有策略，从轻量级 ACT 模型到 SmolVLA 等视觉语言模型

从默认参数开始实验，监控你的动作队列大小，并迭代优化你的设置，以为你特定的用例实现最佳性能。
